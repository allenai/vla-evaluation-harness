# Contributing to the VLA Leaderboard

> **Note on evaluation protocols:** Benchmark evaluation protocols are not fully standardized across the VLA community. Different papers may use the same benchmark name but differ in training regimes, task subsets, or evaluation conditions — making scores not always directly comparable. This leaderboard records all available results transparently and documents known protocol differences.

## Data Structure

The leaderboard data is split across several files:

| File | Contents |
|------|----------|
| `data/benchmarks.json` | Benchmark registry (metrics, suites, tasks, display config) |
| `data/leaderboard.json` | Curated entries (scores, model metadata) — generated by `refine.py` |
| `data/schema.json` | JSON Schema for leaderboard.json |
| `data/citations.json` | Per-paper citation counts from Semantic Scholar |
| `data/coverage.json` | Per-benchmark coverage stats — generated by `extract.py scan` |
| `extractions/{arxiv_id}.json` | Raw LLM extraction per paper (git tracked, incremental) |
| `benchmarks/{key}.md` | Per-benchmark protocol definitions |

### Result Fields

Each result entry in `leaderboard.json`:

```json
{
  "model": "openvla",  "display_name": "OpenVLA",  "params": "7B",
  "model_paper": "https://arxiv.org/abs/2406.09246",
  "benchmark": "libero",  "weight_type": "finetuned",
  "overall_score": 85.7,
  "suite_scores": { "libero_spatial": 84.0, "libero_object": 88.0 },
  "source_paper": "https://arxiv.org/abs/2406.09246",
  "source_table": "Table 1",
  "curated_by": "refine.py (opus)",  "date_added": "2026-04-11"
}
```

| Field | Meaning | Null when |
|-------|---------|-----------|
| `model_paper` | Paper that introduces the model | Proprietary model |
| `source_paper` | Paper where this score was reported | API-synced entry |
| `overall_score` | Aggregate score (controls ranking) | Non-standard protocol |
| `params` | Parameter count (e.g. `"7B"`) | Unknown |
| `name_in_paper` | Exact model label from the source paper (e.g. `"Ours (π₀)"`) | Not yet extracted |

- `weight_type`: `"shared"` (same checkpoint across benchmarks) or `"finetuned"` (trained on this benchmark).
- `overall_score` = `null` when the evaluation protocol doesn't match the benchmark's standard. See `benchmarks/{key}.md` for each benchmark's standard.

## Pipeline

```
extract.py scan    →  discover citing papers (Semantic Scholar)
extract.py run     →  extract scores from papers (LLM, one call per paper)
refine.py          →  deduplicate + curate → leaderboard.json (LLM, opus)
validate.py        →  static checks on leaderboard.json
sync_external.py   →  add API-synced entries (RoboArena, RoboChallenge)
```

### Adding new results

1. `extract.py scan` — find new citing papers
2. `extract.py run --from-scan --workers 2` — extract from new papers
3. `refine.py --model opus` — regenerate leaderboard.json
4. `validate.py leaderboard.json` — check for errors
5. Review diff, commit

### Adding a new benchmark

1. Create `benchmarks/{key}.md` with protocol definition
2. Add benchmark to `benchmarks.json`
3. Re-run extraction on relevant papers: `extract.py run ARXIV_ID ...`
4. `refine.py` will pick up the new benchmark automatically

## Benchmark Protocols

Each benchmark has its own protocol file in `leaderboard/benchmarks/`:

```
benchmarks/
  _global.md          # risky patterns for all benchmarks
  libero.md, calvin.md, ...   # one per benchmark
```

These are the **single source of truth** for benchmark rules. They are:
- Included in `extract.py` LLM prompts for score extraction
- Included in `refine.py` LLM prompts for curation
- The reference for human curators

## Official Leaderboard Policy

Benchmarks with `official_leaderboard` in `benchmarks.json` require API-synced entries only — `curated_by` must end with `-api`.

## Schema

JSON Schema: `data/schema.json`. Key nullable types: `overall_score`, `source_paper`, `source_table`, `params`, `model_paper`, `name_in_paper`.
